Accessibility

How to make sure your service works best for everyone.

Accessibility can refer to many aspects of user experience. Some people use screen readers to read the interface, while other read the interface visually but they struggle with some elements.

Dyspraxia, Dyslexia, ADHD, ADD can all impact the way users experience a website. Mental illness such as depression can affect users’ experience, and in other ways, so can being stressed or being short on time.

Making services work for everyone is an area where content design can really step up and help.

Using plain English goes a long way to making the user experience more accessible. But a content designer should also work to make sure users can access the information, for example, is the page in an accessible format like html? They should also make sure users aren’t overloaded within information they don’t need.

In general, you should keep word count down. The fewer words on a page, the easier a page is to read, and the more likely it is to be read. This is shown in a data visualisation from the Neilson Norman Group.

However, most importantly you should speak to your audience to understand their likely frame of mind when using the service. Test out your ideas to see what works best for your user group.

Best practice advances quickly, but there are some day-to-day things you can do to ensure accessibility isn’t left behind when creating services.

Page titles

Page titles are key to helping people navigate around websites. If they don’t tell the user exactly what’s on the page people may get lost, frustrated, and stressed.

Headers

Make sure pages are structuring using headings and subheadings. We use H1 headings to break pages up, and H2 and H3 headings to create subsections within sections. This makes pages more scannable but it also helps with accessibility since screen readers can easily tab through the page to find the information that users are looking for.

Structure

You should make sure that the order screen readers tab through your pages matches the logical order that users would read the page. You can work with your front-end developer to make sure this is set up correctly.

Aria tags

Use aria codes with the page to create alternative or additional messages that you might want to give to people using a screen reader.

For example, on a page title: 'submit your tax return' there might be a button that says 'Submit now'. This might work fine for users reading the content on a screen, but for people using a screen reader that are tabbing through the page listing to each element in isolation; you might want the aria code for the submit button to be more explicit. For example, 'Submit your tax return now'.

Scalability

You should always check that your designs work on screens of all sizes. Try a browser emulator like Nimbus or other chrome extensions like Window Resizer, or simply resize your browser window manually to see how your content responds when the screen is resized.

Automated testing

You can also use free automated accessibility test to check the overall accessibility of your pages. W3C provide a list of available tools.

Join the accessibility community

A good source of accessibility information is the Government Accessibility Community. It’s very active and there are loads of good threads.

You can also join the Google group and discuss digital accessibility challenges and best practice.
